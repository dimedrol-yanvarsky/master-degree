# Серверная часть

Серверная часть веб-приложения для контроля эмоционального состояния человека.
Реализована на **Go + Gin**, хранилище — **MongoDB**, методология — **Clean Architecture**.

## Архитектура (Clean Architecture)

Зависимости направлены внутрь: внешние слои зависят от внутренних, ядро не знает
о фреймворках и БД.

```
cmd/api                      — точка входа, сборка зависимостей (composition root)
internal/
  entities/                  — Доменные сущности и бизнес-логика (без внешних зависимостей)
    shared/                  — общие примитивы: ошибки, роли, статусы аккаунта
    user, test, recommendation, feedback, collaboration, emotion — сущности подсистем
    support/                 — нечёткий вывод Мамдани (ядро ВКР): фаззификация →
                               активация правил → агрегация → дефаззификация (РПЗ §1.6–1.8)
  usecases/                  — Сценарии использования (Use Cases) и порты (границы)
    port/                    — интерфейсы репозиториев и сервисов (реализуются во внешних слоях)
    support/                 — use case расчёта степени необходимости в поддержке
  adapters/                  — Интерфейсные адаптеры: HTTP, DTO, persistence gateways
    http/                    — Gin-хендлеры, DTO, маршруты, middleware
    persistence/             — MongoDB и in-memory реализации портов репозиториев
  frameworks/                — Внешние детали: драйверы, конфигурация, security, system
    config/                  — загрузка конфигурации окружения
    mongodb/                 — коннектор + обобщённый адаптер к БД (без привязки коллекций)
    security/                — bcrypt-хеширование паролей, выпуск/проверка JWT (HMAC-SHA256)
```

Сетевой фреймворк (Gin) и драйвер БД (MongoDB) — внешние детали инфраструктурного
слоя; бизнес-сценарии работают с ними через интерфейсы (порты), поэтому замена
библиотеки логирования, HTTP-роутера или СУБД не затрагивает доменный и
прикладной слои.

## Запуск

```bash
cd server
cp .env.example .env        # при необходимости поправьте переменные
go run ./cmd/api            # сервер на :8080
```

MongoDB на этом этапе **опциональна**: адаптер доступа к данным подключён, но
коллекции ещё не привязаны, поэтому сервер стартует (и эндпоинт нечёткого вывода
работает) без запущенной БД. Состояние подключения видно в `GET /health`.

## Проверка

```bash
go build ./...

go test ./...              # включает сверку дефаззификации с РПЗ (формула 12 ≈ 61.82)
```

## Эндпоинты

| Метод | Путь                          | Назначение                                  |
|-------|-------------------------------|---------------------------------------------|
| GET   | `/health`                     | Проба живости + статус подключения к БД      |
| POST  | `/api/v1/support/calculate`   | Расчёт степени необходимости в поддержке      |

Пример запроса:

```bash
curl -X POST http://localhost:8080/api/v1/support/calculate \
  -H "Content-Type: application/json" \
  -d '{"extraversion":3.45,"agreeableness":3.22,"conscientiousness":3.52,
       "neuroticism":3.09,"openness":3.83,"emotionalState":44}'
```

Личностные факторы BFI-2 ожидаются на отрезке `[1, 5]`, эмоциональное состояние
(BDS) — на `[16, 64]`; нарушение диапазона → `400`.

## Нечёткий вывод (ядро)

`entities/support` реализует полный цикл алгоритма Мамдани с лингвистическими
переменными и функциями принадлежности из РПЗ (таблицы 14–20):
фаззификация → активация правил (t-норма `min`, s-конорма `max`) →
аккумуляция (`max` по терму) → агрегация (отсечение по силе) →
дефаззификация центром тяжести.

База правил (`defaultRules`) сейчас содержит **документированные примеры**
правил из РПЗ (таблица 21). Полный набор из 59 правил (§1.7) — это доменные
данные, которые подставляются в `NewEngine`/`NewEngineWithRules` без изменения
самого движка.

## Защита от несанкционированного доступа (ТЗ часть 3)

Примитивы реализованы в `frameworks/security`:
- **bcrypt** — хеширование паролей (в БД хранится только хеш);
- **JWT (HMAC-SHA256)** — выпуск и проверка access-токенов на stdlib (без внешних
  зависимостей); в токене только идентификаторы, все решения о доступе
  перепроверяются на сервере.

Middleware `adapters/http/middleware` уже различает коды `401`/`403` через
маппинг доменных ошибок.

## Дорожная карта (следующие итерации)

Слой портов (`usecases/port`) и обобщённый адаптер (`frameworks/mongodb`)
готовы. Дальше подключаются подсистемы по мере финализации коллекций БД:

1. **Учётные записи** — регистрация/авторизация (пароль + Yandex OAuth), сессии,
   восстановление пароля, лимит попыток входа, auth-middleware на защищённых
   маршрутах.
2. **Тестирование** — прохождение тестов, история результатов, управление тестами.
3. **Расчёт поддержки** — оповещение специалиста, запись точки в граф эмоций.
4. **Рекомендательная база**, **отзывы**, **сотрудничество** специалист↔клиент.
5. **Конкретные MongoDB-репозитории** — реализация портов с привязкой коллекций
   согласно даталогической модели (РПЗ рис. 5).
